# 06. UI/UX設計 — 非エンジニアが迷わず使えるために

本アプリの利用者は貿易実務の担当者であり、エンジニアではない。**「システムの使い方を覚える」のではなく「いつもの業務手順を進めると、自然にシステムが完成させてくれる」**体験を設計目標とする。

## 1. 想定ユーザー(ペルソナ)

| ペルソナ | 特徴 | アプリに求めること |
|---|---|---|
| 代表者(管理者) | 商談・価格決定・書類発行を行う。PC中心 | 案件の全体像が一目で分かる。書類を間違いなく速く出せる |
| 業務スタッフ(将来) | 加工・梱包・発送を担当。現場ではタブレット/スマホ | 自分のやるべき作業と入力箇所だけが見える |
| 共通 | Excel・メールは使えるがITは専門外。英語書類は読めるが作るのは負担 | 専門用語の意味がその場で分かる。壊す心配なく触れる |

## 2. 設計原則(5箇条)

1. **次にやることをアプリが教える** — ユーザーが「どの画面で何をすべきか」を考えなくてよい。ダッシュボードと案件画面は常に「次のアクション」を提示する。
2. **選ばせる。書かせない** — 自由入力は最小限。国・通貨・Incoterms・支払条件・加工費説明文はすべて選択式+テンプレート。書くのは数量と金額くらい。
3. **計算と採番は人間にやらせない** — 金額合計・重量合計・容積重量・書類番号はすべて自動。手計算・手入力の余地を残さない。
4. **壊せないUI** — 削除は存在しない(アーカイブのみ)。発行済み書類は編集不可(改訂版発行のみ)。危険な操作は理由の説明付き確認ダイアログ。**「間違えても直せる」ことを画面上に明記**し、触ることへの恐怖をなくす。
5. **UIは日本語、書類は英語** — 画面の項目名・ボタン・警告はすべて日本語。生成されるPDFのみ英語。貿易英語(Incoterms等)は原語のまま表示し、横の「?」アイコンで日本語解説をポップアップ表示する。

## 3. 情報設計(ナビゲーション)

サイドバーは6項目に固定。階層を深くしない。

```
🏠 ホーム(今日やること)
📋 案件
👤 顧客
🍵 商品
🌍 国・輸出ルール
⚙️ 設定
```

- 「書類」「ファイル」「入金」「加工費」は独立メニューにしない。**すべて案件詳細のタブ**として案件に従属させる(実務の思考単位は常に「この案件」であるため)。

## 4. 主要画面のUXパターン

### 4.1 ホーム =「今日やることリスト」

ダッシュボードはグラフではなく、**行動を促すカードの列**にする。

```
⚠️ 対応が必要です
┌────────────────────────────────────────────┐
│ 💰 入金確認待ち 2件                           │
│   DEAL-2026-0012 Green Leaf GmbH  USD 9,430 │
│   PI発行から8日経過 → [入金を記録する]          │
├────────────────────────────────────────────┤
│ 📦 出荷予定日が近い 1件                        │
│   DEAL-2026-0015 出荷予定 7/10               │
│   CI・PLが未発行です → [書類を発行する]         │
├────────────────────────────────────────────┤
│ 🌍 未確認の国からの問い合わせ 1件               │
│   DEAL-2026-0018 ブラジル → [輸出条件を確認]    │
└────────────────────────────────────────────┘
```

### 4.2 案件詳細 = ステータス・ステッパー+「次のアクション」ボタン

11ステータスをそのまま並べると威圧感があるため、**5段階の大ステージに束ねて**表示する。

```
[① 商談] ─ [② 見積・PI] ─ [③ 入金] ─●[④ 加工・出荷] ─ [⑤ 完了]
                                      現在: Repacking(加工中)

  ┌──────────────────────────────────────┐
  │ ▶ 次のアクション: 加工が終わったら         │
  │   [出荷準備完了にする] を押してください      │
  │   その後、CI と Packing List を発行します  │
  └──────────────────────────────────────┘
```

- ステータスは自由に選ばせず、**「次へ進む」ボタン**で順送りが基本(戻す・スキップは「その他の操作」に隔離し、理由メモを求める)。
- ガード条件はボタン上に日本語で表示: 「⚠️ 入金記録がありません。このまま進めますか?」(ブロックせず警告+理由記録 — §05-5)。

### 4.3 書類発行 = 3ステップウィザード

PI/CI/PLの発行は必ず同じ3ステップ。**白紙から書類を作る画面は存在しない**(常に案件データから自動生成した下書きを直すだけ)。

```
Step 1: 内容の確認     — 案件から自動で埋めた内容を表示。修正はその場で
Step 2: 追加情報       — この書類だけに必要な項目(PI: 有効期限 / CI: 追跡番号 / PL: カートン明細)
Step 3: プレビューと発行 — 実物PDFを画面内表示 → [発行する] で番号確定・保存
```

- Step間の移動前に**日本語のチェックリスト検証**を実施し、問題は修正リンク付きで列挙する:
  - 「❌ 発送先住所が空です → 顧客情報を開く」
  - 「⚠️ EU向けなのに買い手のEORI番号がありません → 顧客情報を開く」
  - 「⚠️ 合計重量がPIの数量と合っていない可能性があります」
- 発行後の画面には「修正したいときは: この書類は変更できません。[改訂版を発行]してください」と明記。

### 4.4 国の入力 = 信号機バッジ(§07と連動)

顧客・案件の国フィールドは検索付きセレクトとし、選択と同時に輸出可否バッジを表示。

```
仕向国: [ドイツ ▼]  🟢 輸出可(条件あり: EU農薬基準対応COA必須)
仕向国: [ロシア ▼]  🔴 現在対応不可(制裁・送金不可) — 案件を作成できません
仕向国: [ブラジル ▼] ⚪ 未確認 — 問い合わせ・サンプルまで可。PI発行前に管理者確認が必要
```

### 4.5 入力フォームの共通ルール

| ルール | 内容 |
|---|---|
| 必須は最小限 | 必須は赤アスタリスク付き数項目のみ。それ以外は空で保存可(下書き文化) |
| 自動保存 | フォーム離脱時に「保存されていません」警告。長文フォームは下書き自動保存 |
| 単位を項目内に表示 | 「重量 [____] kg」「単価 [____] USD」— 単位間違いを構造的に防ぐ |
| 例をプレースホルダで | `例: MN-2026-0412`、`例: FOB Hakata` |
| 用語ヘルプ | Incoterms・EORI・HSコード等には ? アイコン → 2〜3行の日本語解説+実務上の注意 |
| 複製ボタン | 「この案件を複製」「前回と同じ加工費を使う」— リピート注文を1分で起票 |
| 日付・金額の表記統一 | 画面は `2026/07/06`・`¥220,000`/`USD 9,430.00`、PDFは英語書式。混在させない |

## 5. ミス防止の仕組み(まとめ)

| リスク | 対策 |
|---|---|
| 金額の計算間違い | 明細×単価・小計・合計はすべて自動計算。合計の手入力欄なし |
| 書類間の数値不一致 | PI/CI/PLは同一案件データから生成。PL発行時にCIとの数量差分を自動チェックして警告 |
| 原価の漏えい | 原価・社内メモはUI上も「🔒 社内のみ」バッジ付きの専用タブに隔離。PDFデータフローから物理分離(§02) |
| 発行済み書類の誤編集 | 発行後は閲覧のみ。Void+改訂版発行フローに誘導 |
| 誤削除 | 削除機能なし。アーカイブ+復元のみ |
| 輸出不可国への販売 | 国マスタ連動でブロック/警告(§07) |
| 入金前の出荷 | ステータスガード+警告(§05-5) |

## 6. モバイル・タブレット対応

- 全画面レスポンシブ。**現場(加工・梱包)で使うのは「QC写真アップロード」「カートン明細入力」「ステータス更新」の3つ**なので、この3機能はスマホで片手操作できることを受け入れ基準にする。
- カメラ起動→撮影→案件のQC写真カテゴリへ直接アップロード。
- タップターゲット44px以上、色だけに頼らない表示(バッジは色+アイコン+文字)。

## 7. 空状態とオンボーディング

- 初回ログイン時、ホームにセットアップチェックリストを表示:
  「① 自社情報を登録 → ② 銀行/Wise情報を登録 → ③ 署名画像を登録 → ④ 最初の商品を登録 → ⑤ 最初の顧客を登録」。完了まで消えない。
- 各一覧の空状態には説明+作成ボタン: 「まだ案件がありません。問い合わせが来たら [案件を作成] から始めましょう」。

## 8. 文言ガイド(抜粋)

- ボタンは動詞で: 「PIを発行する」「入金を記録する」(×「OK」「送信」)
- エラーは「何が・なぜ・どうすれば」の3点セット: 「PIを発行できません。発送先住所が未登録のためです。[顧客情報を開いて登録する]」
- システム都合の用語を出さない: ×「バリデーションエラー」→ ○「入力内容をご確認ください」
