# 08. セキュリティ設計

## 0. 基本姿勢 — 「世界一安全」の正しい解釈

「完璧に安全なシステム」は世界のどこにも存在しない。誠実な設計目標は次の3つである。

1. **多層防御**: 1つの防御が破られても被害が広がらない構造にする。
2. **社内ツールとして過剰でない、しかし省略もしない**: 本アプリが守る資産に見合った、実証済みの標準的対策をすべて実装する。独自の暗号や凝った仕組みは作らない(自作セキュリティは弱点になる)。
3. **技術より運用が先に破られる**: パスワード管理・端末管理・退職時の権限剥奪など、運用ルールを設計に含める。

## 1. 守るべき資産(脅威モデル)

| 資産 | 漏えい/改ざん時の被害 | 重要度 |
|---|---|---|
| 顧客の個人情報(担当者名・メール・住所) | 信用失墜。**EU顧客が含まれるためGDPRの適用対象**になり得る | 高 |
| 取引条件・金額・案件情報 | 競合への漏えい、価格交渉力の喪失 | 高 |
| 自社の銀行/Wise口座情報 | 振込先すり替え詐欺(BEC)の材料に使われる | **最高** |
| 社内原価データ | 利益率の露呈 | 高 |
| 発行済み貿易書類 | 改ざんされると通関・税務トラブル | 高 |
| ログイン認証情報 | 上記すべてへの入口 | **最高** |

特に警戒すべき現実的シナリオは、①メールアカウント乗っ取り等による**振込先すり替え(BEC詐欺)**、②退職者・共有パスワードによる**内部からの持ち出し**、③**依存パッケージ経由の侵害**の3つ。

## 2. 認証(誰が入れるか)

| 対策 | 内容 |
|---|---|
| 招待制のみ | 自己サインアップ機能を実装しない。管理者の招待メールでのみアカウント作成可 |
| **MFA(多要素認証)必須** | Supabase AuthのTOTP(認証アプリ)を全ユーザーに強制。未設定ユーザーはMFA設定画面以外に進めない |
| パスワード | 12文字以上+漏えいパスワード辞書チェック(Supabaseの HaveIBeenPwned 連携を有効化) |
| セッション | JWT短命(1時間)+リフレッシュ。7日操作なしで再ログイン要求 |
| ブルートフォース | Supabase Authの組み込みレート制限を有効化 |
| アカウント削除 | 退職時は管理者が即時無効化(削除でなく無効化 — 監査ログの主体を保持) |

## 3. 認可(入った人が何をできるか)

- **全テーブルでRLS(Row Level Security)有効**。「RLSなしのテーブルはマイグレーションレビューで却下」をルール化。
- 認可判定は必ずDB層(RLS)で行い、UI側の出し分けは利便性のためだけに使う(UIを迂回されても守られる)。
- **ロール権限マトリクス**(将来のスタッフ追加時に有効化。DBスキーマは初期から対応済み):

| リソース | admin | staff | viewer |
|---|---|---|---|
| 顧客・商品・案件・書類・ファイル | 全操作 | 全操作 | 閲覧のみ |
| 原価(`cost_price`)・社内原価メモ(`deal_processing_costs`) | ○ | **不可(RLS+ビューで列ごと遮断)** | 不可 |
| 銀行/Wise情報・自社設定 | ○ | 閲覧のみ | 不可 |
| 国マスタのステータス変更 | ○ | 不可 | 不可 |
| ユーザー招待・ロール変更 | ○ | 不可 | 不可 |

- **Storage**: 全バケット非公開。アクセスは有効期限付き署名URL(60秒)のみ。公開URLは一切発行しない。
- **Supabase service_role キー**はサーバー(Next.jsサーバー側)のみ。クライアントには anon キー+RLSのみ。キーをリポジトリにコミットしない(env管理、§7)。

## 4. データ保護

| 対策 | 内容 |
|---|---|
| 通信 | 全経路TLS(Vercel/Supabase標準)。HSTSヘッダ有効化 |
| 保存時暗号化 | Supabase(AES-256 at rest)標準機能 |
| バックアップ | Supabaseの日次自動バックアップ+**PITR(ポイントインタイムリカバリ)を有効化**。四半期に1回、復元テストを実施(復元できないバックアップは存在しないのと同じ) |
| 書類の不変性 | 発行書類はスナップショット+Void/Rev方式(§02)。改ざん検知として発行時にPDFのSHA-256ハッシュを`documents`に記録 |
| GDPR配慮 | EU顧客の個人データを扱うため: ①収集は業務に必要な項目のみ(既に設計済み) ②削除要請に応えられるよう顧客の匿名化処理(個人名・連絡先のマスク)を将来機能に用意 ③処理目的をプライバシーノートとして文書化 |
| データ最小化 | クレジットカード情報・パスポート等は**そもそも保存しない**(設計にフィールドを作らない) |

## 5. アプリケーション層

| 対策 | 内容 |
|---|---|
| 入力検証 | 全フォーム・全APIでZodスキーマによるサーバーサイド検証(クライアント検証はUX用の補助) |
| SQLインジェクション | Supabaseクライアント/パラメータ化クエリのみ使用。文字列連結SQL禁止 |
| XSS | Reactの標準エスケープに任せ、`dangerouslySetInnerHTML`禁止。CSPヘッダで外部スクリプト遮断 |
| CSRF | Next.js Server ActionsのOrigin検証+SameSite=Lax Cookie |
| セキュリティヘッダ | CSP / X-Frame-Options: DENY / X-Content-Type-Options / Referrer-Policy を`next.config`で全ページ適用 |
| ファイルアップロード | 拡張子+MIMEタイプ+サイズ上限(20MB)検証。実行可能ファイル拒否。ファイル名はUUIDに置換して保存(パストラバーサル防止) |
| PDF生成 | 入力はスナップショットJSONのみ(§03)。外部URL・ユーザー指定パスを読み込まない |
| 依存パッケージ | GitHub Dependabot有効化+月次で`npm audit`。重大脆弱性は即時更新 |
| レート制限 | 書類発行・ファイルアップロードAPIにユーザー単位のレート制限 |

## 6. 監査と検知

```sql
create table audit_logs (
  id          bigint generated always as identity primary key,
  actor_id    uuid references profiles (id),
  action      text not null,       -- 'create' | 'update' | 'archive' | 'issue_doc' | 'void_doc' | 'login' ...
  table_name  text not null,
  record_id   text,
  changes     jsonb,               -- 変更前後の差分(原価・銀行情報はマスクして記録)
  ip_address  inet,
  created_at  timestamptz not null default now()
);
-- INSERTのみ許可(UPDATE/DELETE不可のRLSポリシー)= ログ自体の改ざん防止
```

- 記録対象: ログイン成否、書類発行/Void、ステータス変更、銀行情報・国マスタ・ユーザー権限の変更、ファイルのアップロード/ダウンロード。
- **アラート**(メール/Slack通知): 深夜の大量ダウンロード、ログイン連続失敗、銀行情報の変更(→BEC対策として変更時は全管理者に即時通知)。

## 7. 運用ルール(技術対策とセット)

| ルール | 内容 |
|---|---|
| 環境分離 | 本番/開発でSupabaseプロジェクトを分離。本番データを開発に複製しない(ダミーデータを用意) |
| 秘密情報管理 | APIキー・DBパスワードはVercel環境変数のみ。`.env`はgitignore。リポジトリにキーが混入していないか`gitleaks`等で定期スキャン |
| アカウント原則 | 1人1アカウント。共有アカウント禁止。退職・契約終了当日に無効化 |
| 端末 | 業務端末はOS自動更新+ディスク暗号化+画面ロックを必須化 |
| 振込先すり替え対策 | **自社の銀行/Wise情報の変更は本アプリ上でのみ行い、変更時に全管理者へ通知**。バイヤーには「振込先変更のメールは無効。必ずPI記載の口座のみ有効」と定型文で周知(PIのNotes定型文に含める) |
| インシデント対応 | 「漏えい・乗っ取りを疑ったら: ①全ユーザーのセッション失効 ②キーのローテーション ③audit_logs確認 ④影響顧客への通知」の手順書を用意 |
| 定期点検 | 四半期ごと: ユーザー棚卸し・バックアップ復元テスト・依存更新・国マスタ確認日レビュー |

## 8. リリース前セキュリティチェックリスト

- [ ] 全テーブルRLS有効・ポリシーテスト(別ユーザーで他人のデータが見えないことを自動テスト)
- [ ] 全Storageバケット非公開・署名URL有効期限確認
- [ ] MFA強制が全画面で効いている
- [ ] service_roleキーがクライアントバンドルに含まれない(ビルド成果物をgrep)
- [ ] セキュリティヘッダ適用確認(securityheaders.com でA評価)
- [ ] 未ログインで全URL・全APIが遮断される(自動テスト)
- [ ] staff/viewerロールで原価・銀行情報が取得できない(APIレベルで検証)
- [ ] PITR有効化・復元テスト1回実施
- [ ] audit_logsが主要操作で記録される
- [ ] Dependabot有効・`npm audit`クリーン
