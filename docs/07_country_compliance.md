# 07. 仕向国コンプライアンス設計 — 対応できる国・できない国の管理

さまざまな国から問い合わせが来る前提で、**「この国に売ってよいか・何が必要か」をアプリが即答できる**仕組みを作る。

## 1. 設計の考え方

規制は国ごとに異なり、**頻繁に変わる**。したがってアプリに規制ロジックをハードコードせず、次の構造にする。

1. **国マスタ(destination_countries)** に国ごとの「輸出可否ステータス+要件チェックリスト+最終確認日」を持つ。
2. アプリはそのステータスに応じて**案件フローを制御**する(下表)。
3. 初期データとして本ドキュメントの分類を投入するが、**真実源はJETRO・通関業者・フォワーダーへの確認**であり、確認のたびに国マスタを更新する(最終確認日を記録)。

### ステータスと画面挙動

| ステータス | バッジ | アプリの挙動 |
|---|---|---|
| `ok` 輸出可 | 🟢 | 通常どおり全フロー可 |
| `conditional` 条件付き可 | 🟡 | 案件作成可。国マスタの**要件チェックリストが案件に自動添付**され、全項目チェック完了まで「Ready to Ship」へ進めると警告 |
| `prohibited` 対応不可 | 🔴 | **案件の新規作成をブロック**(理由を表示)。問い合わせ記録用にInquiryステータスのみ許可し、それ以上進めない |
| `unverified` 未確認 | ⚪ | Inquiry / Sample Sent まで可。**PI発行をブロック**し、「管理者が輸出条件を確認して国マスタを更新してください」と誘導 |

> ⚪未確認をデフォルトにすることで、**「調べていない国にうっかり売る」ことが構造的に起きない**。新しい国からの問い合わせは、まず国の調査タスクになる。

## 2. 初期分類(2026年時点の目安 — 取引開始前に必ず最新確認)

**免責的注意**: 以下は設計用の初期値。食品規制・制裁は変動が激しいため、**各国との初回取引前および定期(年1回以上)にJETRO・通関業者へ確認し、国マスタの「最終確認日」を更新する運用**をセットで義務付ける。

### 🟢 輸出可(標準書類で対応可能)

| 国・地域 | 主な要件メモ(国マスタに登録する内容の例) |
|---|---|
| 🇺🇸 米国 | FDA食品施設登録(製造元)、出荷ごとのPrior Notice、買い手側FSVP対応。COA提出を求められるのが通例。関税率は変動があるため都度確認(買い手コスト) |
| 🇬🇧 英国 | 福島関連規制は撤廃済み。買い手のUK EORI必須。日英EPAの原産地申告で関税優遇可 |
| 🇨🇦 カナダ | 輸入者側にSFC(Safe Food for Canadians)ライセンス。CFIA表示規則は輸入者責任 |
| 🇸🇬 シンガポール / 🇭🇰 香港 | 規制軽微。茶は放射性物質関連の対象外。少量取引の立ち上げに最適 |
| 🇨🇭 スイス / 🇳🇴 ノルウェー | 概ねEU準拠。EU向け要件を満たせば対応可 |
| 🇳🇿 ニュージーランド | 加工済み茶は輸入可。バイオセキュリティ申告は通常書類で対応 |

### 🟡 条件付き可(要件チェックリスト必須)

| 国・地域 | 条件(チェックリスト例) |
|---|---|
| 🇪🇺 EU全域 | ①**残留農薬MRL適合のCOA(最重要。茶のMRLは世界最厳格級)** ②買い手のEORI番号 ③日EU EPA原産地申告文で関税ゼロ化(§05-7) ④有機表記する場合はCOI等の有機認証手続 |
| 🇹🇼 台湾 | ①全ての日本産食品に産地証明書 ②特定5県産(福島・茨城・栃木・群馬・千葉)は放射性物質検査報告書も必要(八女産は対象外だが産地証明は必要) ③茶の農薬検査が厳格 |
| 🇰🇷 韓国 | ①一部県産に放射性物質検査証明(福岡産は対象外を都度確認) ②通関時検査が厳しめ。輸入者側の食品輸入登録 |
| 🇦🇺 オーストラリア | ①BICON(輸入条件DB)で茶の条件確認 ②バイオセキュリティ検査対応。加工済み・密封包装の緑茶は概ね可 |
| 🇦🇪 UAE / GCC諸国 | ①輸入者の食品登録 ②アラビア語ラベル(輸入者責任だがOEMでは巻き込まれ注意) ③茶にハラール認証は原則不要だがバイヤー要求あり得る |
| 🇮🇳 インド | ①FSSAI輸入者ライセンス ②茶の関税が極めて高率(100%超)— 価格競争力を要検討 |
| 🇮🇩 インドネシア | ①BPOM登録 ②食品のハラール表示義務化が段階施行中 — 対応可否を要確認 |
| 🇹🇭 タイ / 🇻🇳 ベトナム / 🇲🇾 マレーシア | 輸入者側の食品輸入ライセンス確認。書類は標準+COA |
| 🇧🇷 ブラジル / 中南米 | ANVISA等の輸入者登録が重く、輸送費も高い。案件単位で採算・実現性を確認 |

### 🔴 当面対応不可(案件作成ブロック)

| 国・地域 | 理由 |
|---|---|
| 🇨🇳 中国 | GACC(税関総署)への海外製造者登録が必須+日本産食品への放射性物質全量検査で通関が長期・不確実。**体制構築するまで対応不可扱い**(将来解禁する場合は国マスタを🟡に変更し要件を登録) |
| 🇷🇺 ロシア / 🇧🇾 ベラルーシ | 制裁による決済手段の実質喪失(Wise停止・銀行送金制約)+輸出管理リスク。前払い回収が成立しない |
| 🇰🇵 北朝鮮 | 日本の全面輸出禁止措置 |
| 🇮🇷 イラン / 🇸🇾 シリア / 🇨🇺 キューバ 等の包括制裁国 | 決済不能・制裁リスク。取引しない |

### ⚪ 上記以外のすべての国

デフォルトで「未確認」。問い合わせが来たら: ①JETRO国別情報・通関業者に確認 → ②要件を国マスタに登録し🟢/🟡/🔴を設定 → ③最終確認日を記録 → ④案件を先へ進める。

## 3. データベース追加設計

```sql
create table destination_countries (
  code             text primary key,          -- ISO 3166-1 alpha-2 (例: 'DE')
  name_en          text not null,             -- PDF用 (例: Germany)
  name_ja          text not null,             -- UI用 (例: ドイツ)
  status           text not null default 'unverified'
                   check (status in ('ok', 'conditional', 'prohibited', 'unverified')),
  summary          text,        -- バッジ横の一言(例: EU農薬基準対応COA必須)
  requirements     jsonb not null default '[]',
                   -- 要件チェックリスト。例:
                   -- [{"label": "EU MRL適合COAをLOTに添付済み", "required": true},
                   --  {"label": "買い手のEORI番号を顧客情報に登録済み", "required": true},
                   --  {"label": "日EU EPA原産地申告文をCIに記載(関税ゼロ化)", "required": false}]
  notes            text,        -- 根拠・参照先URL・担当者メモ
  last_reviewed_at date,        -- ★最終確認日。1年超過で一覧に「要再確認」表示
  reviewed_by      uuid references profiles (id),
  updated_at       timestamptz not null default now()
);

-- 案件ごとの要件チェック状況(国マスタのrequirementsを案件作成時にコピー)
create table deal_compliance_checks (
  id          uuid primary key default gen_random_uuid(),
  deal_id     uuid not null references deals (id) on delete cascade,
  label       text not null,
  required    boolean not null default true,
  checked     boolean not null default false,
  checked_by  uuid references profiles (id),
  checked_at  timestamptz,
  sort_order  int not null default 0
);
```

- `deals.destination_country` は `destination_countries.code` への外部キーに変更する(02_database.md 参照)。
- 国マスタの`requirements`を案件作成時に`deal_compliance_checks`へ**コピー**する(後で国マスタを更新しても進行中案件のチェックリストが勝手に変わらない — 書類スナップショットと同じ思想)。
- 全196カ国のISOコード・国名はシードデータとして投入し、上記分類済みの国以外は `unverified` とする。

## 4. 画面

- **国・輸出ルール一覧** (`/countries`): 国名検索、ステータスフィルタ、最終確認日(1年超は⚠️表示)。
- **国詳細/編集** (`/countries/[code]`): ステータス変更、要件チェックリスト編集、確認履歴メモ。ステータス変更時は理由メモ必須。
- 顧客・案件フォームの国セレクトにバッジ表示(§06-4.4)。
- 案件詳細に「輸出要件」タブ: チェックリストの進捗表示、未完了でReady to Shipに進もうとすると警告。

## 5. 制裁・取引先スクリーニング(コンプライアンス)

国単位の判定に加え、**取引相手(企業・個人)単位**の確認も貿易実務上必要になる。

- MVP: 新規顧客登録時に「制裁リスト確認済み」チェックボックス+確認日を顧客に記録(米OFAC SDNリスト・EU制裁リスト・経産省外国ユーザーリストを手動確認する運用)。
- 将来: スクリーニングAPI連携による自動照合(04_roadmap.md に追加)。
- 抹茶は輸出貿易管理令のリスト規制品ではないが、キャッチオール規制の観点から「用途・需要者に懸念がないこと」を確認する建付けにしておく。
